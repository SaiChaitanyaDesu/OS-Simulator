Round Robin (RR) CPU Scheduling Simulation

Explanation

Round Robin (RR) is a preemptive CPU scheduling algorithm where each process is assigned a fixed time slot (quantum) in a cyclic manner. It's designed especially for time-sharing systems.

Key Characteristics:

  • Preemptive: Processes can be interrupted during execution when their time quantum expires.
  • Fair: Each process gets an equal share of CPU time.
  • Time Quantum: A fixed time slice is allocated to each process (typically 10-100ms).
  • Circular Queue: Uses a FIFO queue with processes cycled through repeatedly.

Algorithm Steps:

  1. Maintain a ready queue of processes.
  2. The CPU scheduler picks the first process from the ready queue.
  3. The process runs for the duration of the time quantum or until it completes.
  4. If the process doesn't complete, it's moved to the end of the ready queue.
  5. The scheduler then selects the next process in the queue.

Performance Metrics:

  • Waiting Time (WT): Time a process waits in the ready queue before getting CPU.
  • Turnaround Time (TAT): Total time taken from submission to completion.
  • Average Waiting Time: Sum of waiting times of all processes divided by the number of processes.
  • Average Turnaround Time: Sum of turnaround times of all processes divided by the number of processes.
  • Response Time: Time from submission until the first response is produced.

Turnaround Time = Completion Time - Arrival Time

Waiting Time = Turnaround Time - Burst Time

Response Time = Time of First CPU Allocation - Arrival Time

Sequence of Events (Time Quantum = 2):

  1. P1 arrives at time 0 and runs for 2 units (0-2)
  2. P2 arrives at time 2 and runs for 2 units (2-4)
  3. P1 runs again for 2 units (4-6)
  4. P3 arrives at time 4 and runs for 2 units (6-8)
  5. P4 arrives at time 6 and runs for 2 units (8-10)
  6. P2 runs again for remaining 2 units (10-12)
  7. P1 runs again for remaining 2 units (12-14)
  8. P3 runs again for 2 units (14-16)
  9. P4 runs again for remaining 1 unit (16-17)
  10. P3 runs again for 2 units (18-20)
  11. P3 runs again for remaining 1 unit (20-21)

Advantages:

  • Fair allocation of CPU across all processes.
  • Good for time-sharing systems where response time is important.
  • No process waits indefinitely (no starvation).
  • Better response time than FCFS for interactive systems.

Disadvantages:

  • Performance depends on time quantum: Too large behaves like FCFS, too small increases overhead.
  • Higher average waiting time than SJF for similar processes.
  • Context switching overhead can be significant with small time quanta.
  • Throughput depends on quantum size.

Best Used For:

  • Time-sharing systems where fairness is important.
  • Interactive systems where response time needs to be predictable.
  • General purpose operating systems that need to balance various types of processes.
  • Environments where preventing starvation is more important than optimal throughput.

Time Quantum Considerations:

  • Too large: Approaches FCFS behavior, poor response time
  • Too small: High context switch overhead, reduces throughput
  • Optimal: Typically 10-100ms, should be larger than context switch time
